# 缺陷流程与规范

## 缺陷生命周期

```text
新建
→ 已确认
→ 修复中
→ 待回归
→ 已关闭
```

特殊状态：

- 拒绝：开发或产品认为不是缺陷，需要写明理由。
- 延期：确认是缺陷，但本版本不修复，需要记录风险。
- 重新打开：回归未通过或问题再次出现。
- 重复：与已有缺陷相同，需要关联原缺陷编号。

## 缺陷标题格式

```text
[模块][条件] 实际问题
```

示例：

```text
[购物车][商品库存不足] 仍可将购买数量修改为库存以上
```

## 缺陷证据

根据问题类型收集：

- 页面截图或录屏
- 请求方法、URL、Headers 和 Body
- HTTP 状态码和响应正文
- 数据库查询结果
- 浏览器控制台信息
- 服务端日志和异常堆栈
- 测试账号、商品货号和订单号

## 缺陷提交前检查

1. 是否可以稳定复现？
2. 是否是环境或网络问题？
3. 测试数据是否有效？
4. 是否符合已确认的业务规则？
5. 是否已经存在相同缺陷？
6. 复现步骤是否足够清楚？
7. 预期结果是否有依据？
