## 2. 测试背景

本项目以本机部署的 ECShop B2C 电商系统为测试对象，按照企业测试流程，对用户、商品、购物车、订单及后台管理等核心业务进行验证。

项目不仅执行页面功能测试，还将逐步覆盖接口、数据库、自动化、性能和持续集成，形成可复用、可回归、可展示的完整测试项目。

## 3. 测试目标

1. 验证 Web 前台和后台核心业务功能是否符合需求。
2. 验证游客、注册用户和管理员的权限边界。
3. 验证页面、接口与数据库中的业务数据是否一致。
4. 发现并记录影响用户使用和数据正确性的缺陷。
5. 对稳定、重复执行的核心业务建立自动化回归测试。
6. 对登录、商品查询和订单等关键业务开展性能测试。
7. 输出测试结果、遗留风险和上线建议。

## 5. 测试策略

### 5.1 功能测试

使用以下方法设计用例：

- 等价类划分
- 边界值分析
- 判定表
- 场景法
- 错误推测法
- 状态迁移法

功能测试优先级：

| 优先级 | 说明 | 示例 |
| --- | --- | --- |
| P0 | 系统最核心、阻塞主流程 | 登录、购物车、提交订单 |
| P1 | 重要业务或高频功能 | 注册、搜索、订单查询 |
| P2 | 一般功能和异常体验 | 评论、扩展资料、页面提示 |

### 5.2 接口测试

使用 Postman 完成手工接口验证，重点检查：

- 请求地址和请求方法
- 请求头、Cookie 和请求体
- HTTP 状态码
- 响应正文
- 业务成功标志
- 参数必填、类型、长度和边界
- 身份认证和权限
- 重复请求和幂等性
- 响应时间
- 接口对数据库的影响

### 5.3 数据库测试（后续阶段）

重点验证：

- 注册后用户记录是否正确新增
- 商品页面数据与商品表是否一致
- 购物车数据是否按预期保存
- 订单主表和订单明细是否同时生成
- 订单金额、商品数量和用户归属是否正确
- 后台修改订单状态后数据库和前台是否同步

只执行查询和可控的测试数据操作，不直接修改业务数据来制造“测试通过”。

### 5.4 自动化测试

自动化只覆盖稳定、重要、重复执行的场景：

- 正常登录
- 错误登录
- 商品查询
- 购物车核心操作
- 订单查询
- 核心业务冒烟流程

接口自动化使用：

```text
Python + requests + pytest + 数据驱动 + Allure
```

Web UI 自动化使用：

```text
Python + Selenium + pytest + Page Object + Allure
```

### 5.5 性能测试

使用 JMeter 和 LoadRunner分别开展：

- 基准测试
- 负载测试
- 压力测试
- 峰值测试
- 稳定性测试

主要指标：

- 平均响应时间
- P90、P95、P99
- TPS 或吞吐量
- 错误率
- 并发用户数
- CPU、内存和数据库资源

本机环境结果主要用于学习、比较趋势和发现明显瓶颈，不作为生产系统容量结论。

### 5.6 回归测试

缺陷修复后至少执行：

1. 验证原缺陷是否修复。
2. 验证相关功能是否受到影响。
3. 执行对应模块的核心用例。
4. 对高风险修改执行完整冒烟测试。

## 7. 测试数据策略

### 7.1 数据命名规则

| 数据 | 建议格式 | 示例 |
| --- | --- | --- |
| 用户名 | `qa_阶段_编号` | `qa_func_001` |
| 邮箱 | `用户名@test.local` | `qa_func_001@test.local` |
| 商品 | `[TEST]模块_编号` | `[TEST]空调_001` |
| 收货人 | `测试用户编号` | `测试用户001` |
| 订单备注 | `[QA][日期][用途]` | `[QA][20260625][下单测试]` |

### 7.2 数据分类

- 基础数据：测试开始前已经存在的分类和商品。
- 公共数据：多个用例可以读取，但不能随意修改。
- 独立数据：每条用例使用自己的账号或订单。
- 临时数据：测试结束后可以清理。
- 性能数据：为并发测试批量准备的独立账号。

### 7.3 数据使用原则

1. 不同测试阶段使用不同前缀。
2. 正向用例与异常用例尽量使用独立数据。
3. 性能测试账号数量不少于并发线程数。
4. 不循环复用会产生并发冲突的账号。
5. 对新增数据记录创建时间、用途和清理状态。
6. 不随意删除无法确认归属的数据。

## 8. 准入标准

满足以下条件后，才能正式进入对应阶段：

### 功能测试准入

- ECShop 前台可以正常访问。
- 后台入口可以正常访问。
- MySQL 和 Web 服务运行正常。
- 核心需求和业务流程已经初步明确。
- 测试账号、商品和环境已经准备。
- 测试用例完成并经过检查。

### 自动化测试准入

- 被自动化的功能已基本稳定。
- 手工测试已经通过。
- 元素定位或接口契约相对稳定。
- 测试数据可以重复准备。

### 性能测试准入

- 单用户功能测试通过。
- 业务断言正确。
- 参数化数据数量充足且有效。
- 正式压测时禁用“查看结果树”等高消耗监听器。
- 已明确测试目标和停止条件。

## 9. 准出标准

满足以下条件时，本轮测试可以结束：

- P0、P1 用例全部执行。
- P0 用例通过率为 100%。
- P1 用例通过率达到 95% 以上。
- 不存在未解决的致命或严重缺陷。
- 一般缺陷已有处理结论和风险说明。
- 核心接口和数据库数据验证通过。
- 自动化核心用例能够稳定执行。
- 性能指标达到约定目标，或已明确记录未达标风险。
- 测试报告、缺陷清单和遗留问题已经归档。

## 10. 缺陷管理流程

```text
发现问题
→ 复现并排除环境或数据原因
→ 收集截图、请求、响应、SQL、日志
→ 提交缺陷
→ 开发确认
→ 修复
→ 测试回归
→ 关闭或重新打开
```

缺陷必须包含：

- 缺陷标题
- 所属模块
- 严重程度和优先级
- 测试环境
- 前置条件
- 复现步骤
- 预期结果
- 实际结果
- 证据
- 测试数据
- 日志或数据库信息

## 11. 缺陷等级

| 等级 | 定义 | 示例 |
| --- | --- | --- |
| S1 致命 | 系统不可用、数据严重损坏 | 系统无法启动、订单数据批量丢失 |
| S2 严重 | 核心流程阻塞且无替代方案 | 无法登录、无法提交订单 |
| S3 一般 | 功能异常但不阻塞核心流程 | 搜索结果错误、提示不准确 |
| S4 轻微 | 页面、文字或体验问题 | 错别字、样式轻微错位 |

## 13. 项目风险

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| ECShop 版本较旧 | 技术与当前系统存在差异 | 重点学习通用测试思想和流程 |
| 需求文档缺失 | 容易把推测当成需求 | 建立需求疑问清单并逐项验证 |
| 测试数据污染 | 用例互相影响 | 建立命名规则和数据台账 |
| 本地服务不稳定 | 测试中断或结果失真 | 执行前检查服务、端口和数据库 |
| 第三方服务不可用 | 无法完成真实链路 | 使用 Mock 或记录范围限制 |
| 自动化用例不稳定 | 误报和维护成本增加 | 使用显式等待、独立数据和分层封装 |
| 性能数据不足 | 出现大量业务失败 | 先逐账号验证，再开始并发测试 |
| 本机资源有限 | 性能结论不具生产代表性 | 只做趋势分析并明确环境限制 |

## 当前阶段执行口径

- 注册、登录、购物车、下单等功能优先验证页面交互、业务提示、跳转结果和用户可见状态。
- 缺陷证据优先保留截图、复现步骤、测试数据、请求响应、日志和测试报告。
- 底层数据落库、数据表记录新增、SQL 校验暂不作为当前阶段测试用例的必填步骤。
- 后续进入数据库测试（后续阶段）阶段时，再补充数据库表结构、SQL 校验点和数据一致性检查。
